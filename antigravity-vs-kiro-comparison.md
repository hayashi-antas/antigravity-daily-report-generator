# Antigravity vs Kiro 比較表（実感＋調査）

## 比較表

| 観点 | Google Antigravity | Kiro（spec-driven development） |
| --- | --- | --- |
| 提供元 / 位置づけ | Googleの**agent-first開発プラットフォーム**。IDE＋エージェント管制（Manager）という思想。<br>参考: Google Developers Blog | **spec-driven development（SDD）でAI開発に構造を持ち込む**IDE/CLI。<br>参考: Kiro公式サイト |
| 基本ワークフロー | Planning / Fast 等のモードで「計画 → 実行」を回しやすい設計（計画をArtifactとして扱う思想）。 | **requirements → design → tasks** の流れで、仕様をファイル化しレビューしながら進める。 |
| 生成物の品質（実感） | **Kiroと品質差は感じず**。仕様が明確であれば、普通に完璧に機能した。 | **同様**。慎重に進められる安心感はあるが、最終的な成果物品質に差は体感しなかった。 |
| 速度感 | **速い**。実装に寄せて一気に進めやすい。Planningを挟んでもテンポが良い。 | **慎重**。仕様分解・ドキュメント化・タスク化を重視するため進行は比較的ゆっくり。 |
| 仕様駆動との相性 | 仕様を渡して「計画 → 実装 → 検証」を回しやすい。Planning導線が強い。 | SDDが中心思想。spec（要件・設計・タスク）を**成果物として残す**点が最大の強み。 |
| ドキュメントの残り方 | 計画・タスク・証跡をArtifactとして扱い、コメントでフィードバック可能。 | specがMarkdown等で明示的に残り、レビュー・合意形成・引き継ぎに向く。 |
| 実行権限・自律性 | エディタ・ターミナル・ブラウザまで跨いで実行可能。強力だが運用設計が重要。 | 仕様ファイルによる制御が中心。ルールを明文化した運用に向く。 |
| リスク / ガードレール | 自律実行が強力な分、権限設定・確認ステップが不可欠。実害事例の報道もあり。 | 仕様化・タスク化により手順は明確。ただし操作ミス対策は別途必要。 |
| 使い分け提案（上司向け） | **短時間で動くものを作る / PoC / 試作**に最適。 | **要件を固める / 仕様を資産として残す / 合意形成**が必要な場面に最適。 |

> **Artifactとは**  
> AIが生成した成果物（コード、計画など）を、会話とは独立した編集可能オブジェクトとして扱う概念。  
> 選択箇所にコメントを付けてフィードバックできる。

---

## まとめ

- 同一仕様で比較した結果、**生成物の品質・完成度に有意な差は体感しなかった**（両者とも完璧に動作）。
- 体験としては、**Antigravityは前進が速く**、**Kiroは仕様を丁寧に詰める局面で安心感がある**という性格差が明確。
- 結論として、  
  - **スピード・PoC重視 → Antigravity**  
  - **慎重な合意形成・仕様資産化 → Kiro**  
 という使い分けが合理的。
